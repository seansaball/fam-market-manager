Section 1

Executive Summary

The FAM Market Day Transaction Manager is a desktop application built to support Food Assistance Match operations at farmers' markets. It handles the full transaction lifecycle for each customer visit: recording receipts from vendors, calculating payment breakdowns across multiple assistance programs, applying FAM match benefits, and producing reconciled financial reports.

The system has undergone multiple rounds of thorough review, including automated testing, financial accuracy verification, and full market-day simulations. Every dollar that enters the system has been verified to be correctly tracked, correctly matched, and correctly reported across all outputs.

This system is ready for production use. Financial calculations are precise to the penny. All safeguards are in place to prevent errors, and comprehensive testing confirms the system performs reliably under real-world conditions.

Section 2

What the System Does

The FAM Manager supports the day-to-day financial operations of market events. Here is what it handles:

Market Day Operations

  • Opening and closing market days — each market event is tracked as a distinct session with its own set of transactions and reports.
  • Customer check-in — each customer receives a sequential label (e.g., C-001) for tracking during the day. Returning customers are recognized and handled appropriately.
  • Receipt recording — volunteers enter vendor receipts for each customer, supporting multiple receipts per visit across different vendors.

Payment Processing

  • Multiple payment methods — the system handles SNAP, FMNP, Cash, Credit, Food Rx, and other assistance program payments, each with its own match percentage.
  • FAM match calculation — the system automatically calculates how much FAM contributes based on each payment method's match rate (e.g., 1:1, 2:1).
  • Daily match limits — per-customer caps on FAM match benefits are enforced, including across multiple visits by the same customer.
  • Denomination handling — physical-unit payment methods like FMNP checks ($5 increments) are handled with proper rounding, ensuring the customer is never shortchanged.

Reporting and Reconciliation

  • End-of-day reports — vendor reimbursement totals, customer summaries, and program breakdowns.
  • Google Sheets sync — transaction data syncs to a shared spreadsheet for program-wide visibility.
  • Backup ledger — a plain-text backup file records every confirmed transaction for independent verification.
Section 3

How Financial Accuracy Is Ensured

Penny-Perfect Calculations

One of the most common sources of error in financial software is rounding. When a computer calculates $10.00 ÷ 3, it gets $3.3333... and has to decide what to do with the leftover fraction of a penny. Over many transactions, these fractions can add up to real money going missing or appearing from nowhere.

FAM Manager eliminates this problem by storing all monetary values as whole pennies internally. Instead of "$89.99," the system stores "8999 cents." All math is performed on these whole numbers, which means there are no fractions, no rounding surprises, and no lost pennies.

Verified Totals at Every Step

Every transaction is checked against a simple but critical rule:

What the customer pays + What FAM matches = The receipt total.

This equation must balance exactly for every transaction. The system verifies it automatically and will not allow a transaction that doesn't add up.

Multi-Receipt Accuracy

When a customer shops at multiple vendors in a single visit, each vendor receipt is tracked individually. The system uses a method that guarantees the individual amounts add up to the exact total — no leftover pennies, no shortfalls. The final receipt in the group is calculated as the difference between the total and all prior receipts, making the sum mathematically exact.

Match Limit Precision

When a customer's FAM match is capped at a daily limit (for example, $20), the system distributes the remaining match across payment methods and then verifies the total matches the cap exactly. Any rounding discrepancy of even a single penny is corrected before the transaction is saved.

Section 4

Data Integrity & Reconciliation

Single Source of Truth

The database is the authoritative record of all transactions. Every report, every export, and every synced spreadsheet is generated from the same underlying data. There is no scenario where two outputs show different numbers for the same transaction.

Three-Way Consistency

The system produces three independent outputs, and all three have been verified to agree:

  1. Database — the primary record of every transaction, payment, and match amount.
  2. Backup ledger — a plain-text file that independently records each confirmed transaction. This serves as a human-readable backup that can be checked by hand if needed.
  3. Google Sheets sync — transaction data pushed to a shared spreadsheet for reporting and program oversight.

Testing has confirmed that for every transaction — including complex scenarios with multiple payment methods, match caps, and denomination rounding — all three outputs show identical totals.

Voided Transactions Are Fully Excluded

When a transaction is voided (cancelled), it is completely removed from all financial calculations. Voided transactions do not appear in vendor reimbursement reports, do not count toward daily match limits, and do not sync to external spreadsheets. This has been verified across every reporting pathway in the system.

Section 5

Safeguards and Controls

Input Validation

  • Receipt amounts must be greater than zero.
  • A vendor must be selected before a receipt can be added.
  • Payment amounts are capped to prevent exceeding the order total.
  • Denomination-based payments (like FMNP checks) only allow valid increments.
  • Zip codes are validated for proper format when entered.

Transaction Safety

  • All-or-nothing saves — when a transaction is confirmed, all related records (the transaction, payment details, match amounts, and audit log) are saved together as a single unit. If anything fails partway through, everything is rolled back. There is no possibility of a "half-saved" transaction.
  • Double-click protection — the confirm button is disabled immediately after the first click, preventing duplicate submissions.
  • Draft mode — transactions start as drafts and are only finalized when the volunteer explicitly confirms. This allows review and correction before commitment.

Corrections and Adjustments

  • Edit capability — confirmed transactions can be adjusted if an error is discovered (wrong amount, wrong vendor, etc.). The adjustment is tracked in the audit log.
  • Void capability — transactions can be voided entirely. Voided transactions are preserved for the audit trail but excluded from all financial totals and reports.
  • Customer order void — an entire customer visit can be voided at once, cancelling all associated receipts and payments.

Audit Trail

Every significant action — creating a transaction, confirming it, adjusting it, or voiding it — is logged with a timestamp and description. This provides a clear history of what happened and when, which is valuable for accountability and troubleshooting.

Backup Ledger

In addition to the database, a plain-text backup file records every confirmed transaction in a human-readable format. If the database were ever damaged or questioned, this ledger provides an independent record that can be reviewed line by line.

Section 7

Risks and Mitigations

No system is without risk. The following table identifies the known risks, their severity, and what is in place to address them.

Risk Severity Mitigation
Human data entry error
(wrong amount or vendor)		 Low Volunteers review transactions before confirming. Adjustments and voids are available after the fact. Audit log records all changes. This is an operational training matter, not a software defect.
Computer crash during a transaction Low All financial writes use an all-or-nothing approach. If the computer loses power mid-save, the database rolls back to the last complete state. No partial or corrupted transactions are possible.
Database file damage
(hardware failure, accidental deletion)		 Low The backup ledger provides an independent plain-text record of all confirmed transactions. Automatic backups of the database are also created. The database uses a crash-safe storage mode designed to survive unexpected shutdowns.
Google Sheets sync failure
(no internet at market)		 Low The database and backup ledger are stored locally and do not require internet. Google Sheets sync can be retried later. No financial data is lost if the sync fails.
Running two copies simultaneously Prevented The application detects if another copy is already open and displays a warning instead of launching a second instance. This prevents conflicting writes to the database.

Note: No high-severity or unmitigated risks were identified during this assessment. The risks listed above are standard operational considerations common to any transaction management system, and all have appropriate safeguards in place.

Section 8

Operational Readiness

How the System Will Be Used

On market day, a volunteer opens the application and selects (or creates) the current market day. As customers arrive, the volunteer:

  1. Creates a new customer order (the system assigns a label like C-001).
  2. Enters vendor receipts — one per vendor the customer shopped at.
  3. Selects payment methods and enters the amounts tendered.
  4. Reviews the payment breakdown, including the FAM match amount.
  5. Confirms the transaction.

At the end of the day, the volunteer can review reports showing vendor reimbursement totals, customer summaries, and program totals. Data syncs to Google Sheets for program-level reporting.

What Volunteers Need to Know

  • Enter receipt amounts carefully — the system validates inputs but relies on accurate data entry.
  • Review the payment breakdown before confirming.
  • Use the adjustment or void feature if a mistake is discovered after confirmation.
  • Close the market day at the end of operations to finalize reports.

Day-to-Day Confidence

The system has been designed to be forgiving of real-world conditions: it handles corrections gracefully, prevents common mistakes through input validation, and keeps a complete audit trail. Volunteers do not need technical expertise to operate it effectively.
