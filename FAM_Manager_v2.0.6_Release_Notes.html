FAM Market Manager v2.0.6
Release Summary  ·  May 2026

v2.0.6 covers everything since v1.9.8 (April 27, 2026). The headline additions are per-vendor payment-method eligibility, a configurable rewards engine, and a redesigned Payment Confirmation Dialog. Behind the scenes, multi-workstation cloud sync is now systematically guarded — every local mutation that affects a cloud-bound row is wired to a sync trigger, and every cleanup respects per-device ownership.

Schema: v34  ·  Tests: 3,387 (up from 1,822 at v1.9.8)  ·  Migration path: automatic on first launch, with pre-migration backup

What's New for Volunteers & Coordinators

Per-Vendor Payment-Method Eligibility

Configure, per vendor, exactly which payment methods that vendor can accept (Settings → Vendors). Ineligible methods no longer appear as options against ineligible vendors — a baked-goods booth can't accidentally take Food Bucks. Mid-season changes propagate to every laptop on the next sync.

Configurable Rewards Engine

Rules of the form "For every $X spent on [source method], hand out N units of [reward method] worth $Y each." Default is the classic $5 SNAP → 1 × $2 JH Food Bucks, but every part is editable. The new Generated Rewards report tab lists every reward issued for end-of-season reconciliation against your token inventory.

Redesigned Payment Confirmation Dialog

A marching-ants action zone walks the volunteer through the physical steps before the transaction commits. Per-method action rows ("Collect $10.00 SNAP", "Hand over 2 × $2.00 Food Bucks tokens"). Required acknowledgment checkboxes for SNAP/EBT.

Reports & Analytics

  • Zip Code column on Detailed Ledger, Transaction Log, Generated Rewards, and FMNP Entries (internal + Sheets).
  • Vendor Reimbursement rebuilt with accurate per-vendor totals and a separate FMNP External column.
  • FAM Match vs FAM Absorbed as two distinct columns.
  • Activity Log with full audit trail across every CREATE / CONFIRM / ADJUST / VOID action.

Multi-Workstation Cloud Sync — Hardened

Pre-release end-to-end testing (May 2026) surfaced a class of cases where local mutations didn't always reach the shared Google Sheet, or reached it in ways that conflicted across co-located workstations. v2.0.6 closes those gaps.

  • Settings changes propagate. Vendor / market / payment-method add, edit, toggle, and delete; the four assignment dialogs; reward-rule changes — all now trigger a full-scope cloud sync.
  • Closed market day mutations sync. FMNP entries added to closed days and Admin adjustments / voids on historical receipts now scope to the affected day, not the currently-open one.
  • Vendor Reimbursement cleanup is multi-market-aware. Stale rows owned by this device are now removed across every market it's worked, not just its currently-configured primary.
  • Reset preserves other devices' data. Resetting a local app instance now triggers an immediate sync that clears only this device's rows from the shared sheet — other workstations' data is untouched.
  • Market renames protected. Renames that would shift the derived market_code are blocked once the market has any history; code-stable renames (typo fixes) are always allowed.
  • Photo dedup cache cleaned on void / delete / replace. Re-uploading the same image to a fresh transaction or FMNP entry now triggers a fresh Drive upload instead of pointing at a VOID-renamed file.

Reliability & Diagnostics Polish

  • Single-instance lock against two app instances on one database.
  • Pre-migration database snapshot on every schema upgrade — failure to write the backup blocks the migration entirely.
  • Per-line invariant trigger ensures every payment-method row satisfies customer_charged + match = method_amount.
  • Voided is terminal — DB-trigger enforced.
  • Atomic ledger backup on every confirm / adjustment, with 5-snapshot rotation.
  • Error Log tab auto-refreshes on every selection so the local view stays in sync with the cloud Error Log sheet.
  • Fragmentation hint is size-aware — only fires on databases >4MB with >30% reclaimable space, silent on small/test installs.
  • Device-identity dialog is actionable — image-cloned laptops with a missing MachineGuid get clear PowerShell remediation instead of a generic error.

Upgrading from v1.9.8

In-app update (recommended): open the app, go to Settings → Updates, click Check for Updates, then Download & Install. The app handles the rest: app folder is backed up, new version is extracted, integrity-verified rollback runs if anything goes wrong.

Manual update: download the zip from the GitHub release page, replace the old application folder, re-launch.

Data safety: existing data is preserved automatically. Cloud credentials, Drive folder, and update channel settings survive the upgrade. The schema migration runs on first launch, with a pre-migration backup written before any change.

What you'll see on existing Sheets: a new Zip Code column appears on Detailed Ledger, Transaction Log, Generated Rewards, and FMNP Entries on the next sync. Historical rows leave it empty; new rows from v2.0.6 onward populate it from each customer's order.

Pre-flight check for image-cloned fleet laptops

If your fleet was deployed by cloning one Windows image across multiple laptops, take 30 seconds to verify each device has its own MachineGuid before upgrading. v2.0.6 refuses to launch on devices that share a synthetic device identity (this prevents silent cross-device sync corruption). In an elevated PowerShell on each laptop: 

(Get-ItemProperty 'HKLM:\SOFTWARE\Microsoft\Cryptography' MachineGuid).MachineGuid

You should see a unique GUID per device. If a value is missing, blank, or identical across laptops, generate a fresh one before launching v2.0.6: 

$g = [guid]::NewGuid().ToString()
Set-ItemProperty 'HKLM:\SOFTWARE\Microsoft\Cryptography' MachineGuid $g

If a device hits this on first launch of v2.0.6, you'll see a Device Identity Required dialog with the same instructions — no data is touched, and the device runs normally once the registry value is set.
